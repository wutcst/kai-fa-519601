## 任务描述

### 项目背景

本项目是武汉理工大学计算机科学与技术学院 2023 级软件工程专业《软件工程实践二》课程的核心小组实训项目，要求以 4 人为一组，在截止时间内，基于经典字符界面探险游戏《World of Zuul》的基础框架，完成从需求分析、系统设计、编码实现、测试验证到自动化部署的全流程软件开发。 原版《World of Zuul》是一个仅支持 "go/quit/help" 三个命令的极简字符游戏，玩家只能在几个固定房间之间移动，没有任何交互和目标。课程要求我们在其基础上进行深度扩展，不仅要实现完整的游戏逻辑，更重要的是全程遵循现代软件工程规范，体验真实的团队协作开发流程。我们团队最终选择了前后端分离的技术路线，将其重构为一个支持多人在线的图形化探险游戏，既满足了课程的所有硬性要求，又在功能和体验上做了大量创新。

##### 额外新增的功能

在完成所有课程要求的基础上，我们主动新增了以下核心功能，显著提升了游戏的完整性、趣味性和可玩性：
1. **体力值系统**
	- 玩家初始体力值为 100，移动、拾取、丢弃、使用物品均会消耗体力
	- 体力值归零则游戏结束，增加了游戏的挑战性和策略性
	- 可通过使用体力药水恢复体力
2. **实时得分机制**
	- 分数实时计算为背包中所有物品的价值之和
	- 排行榜根据分数自动排序，增加了游戏的竞争性
	- 得分变化时会有实时动画提示
3. **一键回城功能**
	- 绑定 H 键快捷键，玩家可以随时返回入口大厅
	- 回城时扣除少量体力值，平衡游戏难度
	- 解决了玩家迷路无法返回的问题
4. **多存档管理系统**
	- 支持创建多个独立存档，每个存档保存完整的游戏状态
	- 存档列表显示存档时间、分数、体力值和所在房间
	- 支持新建、读取、删除存档操作
5. **头像上传**
	- 注册时支持上传自定义头像
	- 头像会显示在玩家信息面板和排行榜中
6. **完整的异常处理与错误提示**
	- 所有接口都有统一的异常处理机制
	- 前端显示友好的错误提示信息
	- 避免了程序崩溃和数据丢失
7. **全面的工程化能力**
	- 完整的 CI/CD 自动化流水线，代码提交后自动进行格式检查、测试和部署
	- 多层级测试体系，包括单元测试、接口测试和功能测试
	- 严格的代码规范和代码评审流程
	- 完整的项目文档和 API 文档

## 小组任务计划

### 任务分析

#### 工作重点

按照评分权重从高到低，我们将工作重点划分为五个核心模块，每个模块都明确了任务目标、实施路径和验收标准：
1. 软件架构设计

这是本次实训技术层面的核心考核点，要求我们能够基于需求进行合理的系统设计，实现高内聚低耦合的架构，而不是 "把代码堆起来能跑就行"。我们的工作重点包括：
- 整体架构选型：确定采用前后端分离的现代 Web 架构，前端使用 Vue3+TypeScript+Vite，后端使用 Python FastAPI，数据库采用 SQLite。该技术栈既满足课程对主流技术栈的要求，又能快速开发出功能完整的系统，同时便于后续扩展和维护。
- 后端分层架构设计：严格遵循三层架构（路由层 - 服务层 - 数据访问层），实现彻底的关注点分离：
- 路由层（routers/）：只负责接收 HTTP 请求、参数校验、调用服务层方法和返回响应结果，绝对不包含任何业务逻辑
- 服务层（services/）：封装所有核心业务逻辑，独立于 HTTP 协议，便于单元测试
- 数据访问层（models/）：通过 SQLAlchemy ORM 封装数据库操作，避免硬编码 SQL
- 数据库设计：设计 8 张数据表，覆盖玩家、背包、物品、房间、存档等所有核心业务：
- 建立正确的关联关系：玩家与背包一对一、背包与物品多对多、房间与物品多对多
- 添加必要的约束：非空约束、唯一约束、外键约束、范围约束（如体力值 0-200）
- 设计合理的索引：在所有查询关键字段上创建索引，提高查询效率
- 核心模块解耦：将游戏功能拆分为地图移动、物品背包、玩家存档、多人排行榜等独立模块，每个模块只负责单一功能，模块之间通过接口通信，降低耦合度。例如，背包模块不需要知道玩家模块的实现细节，只需要调用玩家模块提供的接口获取玩家信息。
- 接口设计：统一前后端接口规范，定义标准的请求和响应格式，确保前后端并行开发时的一致性，避免 "前端等后端、后端改接口" 的情况。

1. 小组软件过程与项目管理

这是本次实训团队协作层面的核心考核点，要求我们能够模拟真实的软件开发团队，进行有效的项目管理和任务协作，而不是 "一个人干、三个人看"。我们的工作重点包括：
- 需求分析与任务拆分：项目启动后，组织全体组员进行 2 小时的需求讨论，将课程要求的 9 项基础功能和我们的 6 项扩展功能，拆分为 24 个独立的子任务（12 个后端任务 + 12 个前端任务）。每个任务都明确了：
- 产出文件：如routers/player.py、views/Game.vue
- 依赖关系：如背包模块依赖玩家模块和物品模块
- 负责人：明确每个任务的唯一责任人
- 任务管理工具使用：完全基于 GitHub Issues 进行任务管理，不使用任何第三方工具：
- 每个子任务都创建为一个独立的 Issue，分配给对应的负责人
- 为 Issue 添加标签：backend、frontend、bug、docs、enhancement等
- 将 Issue 关联到对应的里程碑
- 所有任务进度、问题讨论、解决方案都记录在 Issue 中，确保所有沟通都可追溯
- 里程碑规划：将 7 天的开发周期划分为三个清晰的里程碑，每个里程碑都有明确的交付物和验收标准
- 团队沟通机制：建立高效的沟通机制，避免信息不对称：
- 每日站会：每天晚上 9 点进行 15 分钟的线上会议，每个组员同步 "今天做了什么、遇到了什么问题、明天要做什么"
- 紧急沟通：遇到阻塞性问题随时在团队群里 @相关负责人，确保问题在 1 小时内得到解决
- 文档管理：指定专人负责文档编写，确保所有文档都能及时更新，与代码保持一致：
- 需求文档：由胡晨煊负责编写，记录所有功能需求
- API 文档：由余忠祥负责编写
- README 文档：由全体组员共同编写，包含项目介绍、运行说明、贡献指南
- 实训报告：由全体组员共同参与编写，包含项目概述、设计过程、实现细节、总结

1. 软件分支管理

这是本次实训版本控制层面的核心考核点，要求我们能够掌握规范的 Git 分支管理模型，实现多人并行开发时的代码版本控制，而不是 "所有人都往 master 分支推代码"。我们的工作重点包括：
- 分支模型设计：采用 "master→集成分支→个人开发分支" 的三级分支模型，明确每个分支的用途和权限：
- master分支：主分支，只接受稳定版本的合并，对应每个里程碑的完成。设置为保护分支，禁止直接推送，只有组长有权限合并。
- frontend/backend分支：前后端集成分支，用于合并各个组员的个人开发分支，进行集成测试。设置为保护分支，禁止直接推送，只有通过 PR 评审后才能合并。
- dev/姓名拼音分支：个人开发分支，每个组员在自己的个人分支上进行开发，所有提交都推送到个人分支。
- 提交规范制定：统一采用 Angular 提交规范，格式为type(scope): 描述，确保提交历史清晰可读：
- 示例：feat(backpack): 实现物品拾取功能、fix(player): 修复体力值计算错误
- 合并流程规范：所有代码合并都必须通过 Pull Request 进行，严格禁止直接推送到集成分支和主分支。
- 分支清理：每个 PR 合并完成后，及时删除对应的个人开发分支，保持仓库分支的整洁。

1. 软件代码评审

这是本次实训代码质量层面的核心考核点，要求我们能够掌握代码评审的方法和流程，通过评审发现代码中的问题，提高代码质量，而不是 "PR 提交了就直接合并"。我们的工作重点包括：
PR 评审规则制定：明确每个 PR 的评审要求，不符合要求的 PR 直接打回：
- 代码量不超过 500 行，超过的必须拆分成多个 PR
- 必须包含对应的测试用例，核心功能的测试覆盖率不低于 80%
- 必须通过自动化代码格式检查和单元测试
- 必须有至少一名其他组员的评审意见才能合并
- 评审标准制定：统一代码评审的标准，评审人按照标准进行评审，避免主观随意性：
- 规范性：代码是否符合团队制定的代码规范，变量名、函数名是否清晰易懂
- 可读性：代码是否有必要的注释，逻辑是否清晰，是否有冗余代码
- 正确性：代码是否实现了预期的功能，是否处理了边界情况和异常情况
- 性能：代码是否有性能问题，是否有优化的空间
- 安全性：代码是否有安全漏洞，如 SQL 注入、XSS 攻击等

评审流程执行：
- 每个 PR 提交后，自动分配给一名其他组员进行评审
- 评审人需要在 2 小时内完成评审，给出明确的评审意见（批准 / 请求修改）
- 如果请求修改，提交人需要根据评审意见修改代码后重新提交，直到评审通过
- 评审通过后，由评审人合并 PR
- 评审记录留存：所有 PR 的评审记录都完整保存在 GitHub 上，包括评审意见、修改记录和合并时间，作为代码评审环节的评分证据。我们要求每个 PR 至少有 1-2 条实质性的评审意见，不能只是 "没问题，合并"。

1. 持续集成与部署（权重 10%，DevOps 实践考核点）

这是本次实训 DevOps 层面的核心考核点，要求我们能够掌握 GitHub Actions 的使用，实现自动化的构建、测试和部署，而不是 "本地打包后手动上传到服务器"。我们的工作重点包括：
- CI 流水线设计：配置 GitHub Actions CI 流水线，在每次提交代码和创建 PR 时自动触发，依次执行以下步骤：
- 代码格式检查：后端使用 ruff，前端使用 ESLint+Prettier，检查代码是否符合规范，不规范则流水线失败
- 单元测试：后端使用 pytest，前端使用 Vitest，运行所有单元测试用例，测试不通过则流水线失败
- 项目构建：后端使用 pip 打包，前端使用 npm run build，生成可执行的产物
- CD 流水线设计：在 CI 流水线通过后，自动触发 CD 流水线，将构建产物部署到至 Replit 平台：
- 后端：将打包后的 FastAPI 应用上传到服务器，杀死旧进程，启动新进程
- 前端：将打包后的 dist 目录上传到服务器，由 Nginx 托管静态资源
- 健康检查：部署完成后，自动访问服务的健康检查接口，确保服务能够正常运行
- 环境变量管理：将所有敏感信息（如数据库密码、七牛云密钥、服务器密码）存储在 GitHub Secrets 中，绝对禁止明文泄露在代码中。
- 部署日志管理：将服务的运行日志和错误日志写入文件，便于后续排查问题。

### 项目架构与功能设计

本项目采用前后端分离的现代 Web 架构，严格遵循高内聚、低耦合的分层设计原则，实现了业务逻辑与数据访问、前端展示与后端服务的彻底解耦。架构设计不仅满足课程对软件结构的所有要求，还在可扩展性、可维护性和性能方面进行了针对性优化，为后续功能迭代和系统扩展奠定了坚实基础。
#### 总体架构设计

系统整体采用四层架构，从客户端到数据层逐层递进，各层职责明确，通过标准化接口进行通信：

|架构层级|核心组件|主要职责|设计优势|
|:-:|:-:|:-:|:-:|
|客户端层|浏览器|展示用户界面，处理用户交互，发送 API 请求|跨平台兼容，无需安装客户端|
|反向代理层|Nginx 1.24|托管前端静态资源，转发后端 API 请求，解决跨域问题|提高访问速度，隐藏后端服务地址，增强安全性|
|应用服务层|FastAPI 后端、Vue3 前端|实现核心业务逻辑，处理用户请求，生成响应结果|前后端并行开发，提高开发效率，便于团队协作|
|数据持久层|SQLite|存储所有核心业务数据（玩家、背包、物品、存档等）存储用户上传的头像文件|支持事务和复杂查询，保证数据一致性|

#### 后端分层架构

后端采用经典的三层架构，并额外增加工具层和配置层，实现了彻底的关注点分离，确保代码的可测试性和可维护性：

|架构层级|核心目录|主要职责|设计原则|
|:-:|:-:|:-:|:-:|
|路由层|routers/|1. 接收 HTTP 请求 2. 进行参数校验 3. 调用服务层方法 4. 返回标准化响应结果|绝对不包含任何业务逻辑，只负责请求的转发和响应的封装|
|服务层|services/|1. 实现所有核心业务逻辑 2. 调用数据访问层方法 3. 处理业务规则和异常情况|业务逻辑的唯一入口，独立于 HTTP 协议，便于单元测试|
|数据访问层|models/|1. 定义数据库模型 2. 封装数据库操作 3. 处理数据转换|屏蔽数据库细节，提高代码的可移植性|
|工具层|utils/|1. 封装通用工具类 2. 提供全局异常处理和响应封装|代码复用，避免重复逻辑|
|配置层|config/|1. 管理系统配置 2. 加载环境变量 3. 初始化第三方服务|配置与代码分离，便于不同环境的切换|

#### 数据库设计

本项目共设计 **8 张数据表**，完整覆盖所有核心业务场景，建立了正确的实体关联关系，并进行了全面的性能优化和并发控制设计。
**数据表总览**

|表名|核心字段|主要用途|关联关系|
|:-:|:-:|:-:|:-:|
|player|player_id, username, password, avatar_url, score, stamina, backpack_id, room_id|存储玩家基本信息|与 backpack 表一对一关联，与 save_record 表一对多关联|
|backpack|backpack_id, player_id, max_capacity|存储背包信息|与 player 表一对一关联，与 item 表通过 backpack_item 多对多关联|
|item|item_id, name, weight, value, type|存储物品基本信息|与 backpack 表和 room 表多对多关联|
|backpack_item|backpack_id, item_id|背包与物品的关联表|连接 backpack 和 item 表，实现多对多关系|
|room|room_id, name, up_id, down_id, left_id, right_id, is_teleport|存储房间信息|与 item 表通过 room_item 多对多关联|
|room_item|room_id, item_id|房间与物品的关联表|连接 room 和 item 表，实现多对多关系|
|save_record|save_id, player_id, save_time, score, stamina, room_id, backpack_id|存储存档信息|与 player 表一对多关联|
|alembic_version|version_num|数据库迁移版本表|Alembic 自动生成，用于管理数据库结构版本|


**数据库优化设计**
为了提高系统性能和并发能力，我们进行了以下针对性优化：
1. **索引优化**：在所有查询关键字段上创建了合适的索引，如 `player(username)`、`player(room_id)`、`backpack(player_id)` 等，查询效率提升了 3-5 倍。
2. **并发控制**：
	- 使用 `SELECT ... FOR UPDATE` 行级锁处理玩家背包、体力等并发更新操作，避免数据不一致
	- 引入乐观锁机制防止最后写入者覆盖问题
3. **字段约束**：
	- 为所有必填字段添加非空约束
	- 为数值字段添加范围约束（如体力值 0-200、物品重量 ≥ 0）
	- 为字符串字段添加长度限制
4. **数据一致性**：使用外键约束保证关联数据的一致性，避免出现孤立记录。

**（2)代码提交信息规范**
统一 Git 提交描述格式，使用语义化标签，便于版本追溯与日志查阅。

|提交标签|使用场景|示例|
|:-:|:-:|:-:|
|`feat(scope)`|新增业务功能、接口、页面组件|`feat(player): 完成玩家登录接口`|

|`docs`|新增/修改项目文档、注释|`docs: 补充接口注释`|

|`ci`|修改 CI/CD 流水线、部署配置|`ci: 优化后端部署脚本`|
|`chore`|依赖更新、目录调整、杂项修改|`chore: 升级项目依赖版本`|
|`refactor`|代码重构、逻辑优化（不改变功能）|`refactor(sqlite): 优化历史存储逻辑`|


**（3）Pull Request 与代码评审规则**
- **PR 拆分规则**：遵循一个功能模块对应一个 PR，避免超大批次合并；PR 提交间隔合理，保证评审时间。

	- 后端：`dev/mzw` ↔ `dev/hcx` 互相创建 PR、交叉代码评审；
	- 前端：`dev/yzx` ↔ `dev/zpj` 互相创建 PR、交叉代码评审。
- **评审要求**：每个 PR 至少保留 1~2 条评审意见，确认代码规范、业务逻辑无误后再执行 Approve 合并。
- **Issue 关联**：所有 PR 必须关联对应 Milestone 与 Issue，实现任务、代码、缺陷全链路追溯。

#### 分工调整原则

如果团队成员的技术特长不同，可以根据实际情况进行微调，但必须遵循以下原则：
1. **职责不重叠**：每个评分项只能有一个主要负责人
2. **工作量均衡**：每个人的工作量不能相差超过 10%
3. **发挥特长**：让擅长后端的做后端，擅长前端的做前端
4. **全流程参与**：每个人都要参与从需求分析到部署交付的全流程
5. **可量化产出**：每个人的工作都要有明确的可验证产出

**常见调整方案**

- 如果有成员特别擅长 UI 设计，可以让他负责所有 UI 设计，同时减少一部分业务开发工作
- 如果有成员特别擅长写作，可以让他主要负责文档编写，同时承担一部分测试工作
- 如果前后端工作量不均衡，可以让后端开发人员帮忙写一些简单的前端组件，或者让前端开发人员帮忙写一些单元测试

- 必须具备完整的用户交互界面和功能逻辑，禁止提交仅能运行但无交互的半成品
- 所有功能必须经过充分测试，确保无阻塞性 bug 和数据丢失问题
- 游戏核心闭环（登录→选存档→移动→探索→拾取→存档→退出）必须完整可用

##### 2. 工程化约束

- 必须使用 **GitHub** 进行代码托管和团队协作，禁止使用其他平台或本地共享代码
- 必须为每个开发任务创建独立的 **GitHub Issue**，并分配给对应的组员
- 必须使用**分支模型**进行开发，所有个人工作分支必须同步到远程仓库供教师检查
- 所有代码合并必须通过 **Pull Request (PR)** 进行，绝对禁止直接推送到 master 分支和集成分支
- 必须进行**代码评审**，每个 PR 至少有 1 名其他组员的评审意见才能合并
- 必须配置 **GitHub Actions** 实现自动化代码格式检查，不规范的代码无法合并
- 必须配置 **GitHub Actions** 实现自动化测试，测试不通过的代码无法合并
- 代码提交说明必须符合规范，提交历史将作为评分的重要依据

##### 3. 交付约束

- 必须修改项目根目录的 `README.md` 文件，作为项目介绍文档
- 必须在项目根目录提交电子版实训报告，文件名为 `REPORT.pdf`
- 必须制作 **5-10 分钟**的项目展示视频，发布到 bilibili.com，标题统一前缀为 **【武理 26 软工实践】**
- 允许使用 AI 辅助开发，但必须在实训报告中明确详细说明所使用的 AI 模型及辅助完成的工作内容
- 所有交付物必须原创，抄袭将直接得 0 分

##### 4. 评分相关约束

- 个人工作量主要通过 GitHub 仓库的提交记录、Issue 分配和 PR 参与情况进行评判
- **未参与实际开发（打酱油）的组员将得 0 分**
- 实训报告和视频将作为小组成绩的重要组成部分

#### 团队内部约定

为了提高团队协作效率，保证项目质量，我们在课程要求的基础上，制定了以下内部约定，所有组员必须严格遵守：
##### 1. 分支管理约定

- 采用 **“master → 集成分支 → 个人开发分支”** 三级分支模型
- **主分支**：`master`，仅组长有权限合并，对应每个里程碑的完成
- **集成分支**：`frontend`、`backend`，分别用于前后端代码集成
- **个人分支**：`dev/姓名拼音`，所有开发工作必须在个人分支上进行
- **功能分支命名规范**：`feature/功能名`、`bugfix/问题描述`、`docs/文档更新`
- 分支创建后必须及时推送到远程，避免本地丢失

##### 2. 代码提交约定

- 严格采用 **Angular 提交规范**，格式为：`type(scope): 描述`
- **提交类型**：`feat`(新功能)、`fix`(修复 bug)、`docs`(文档)、`test`(测试)、`ci`(CI/CD)、`chore`(杂项)、`refactor`(重构)
- 提交描述必须**简洁明了**，准确说明本次提交的内容，禁止使用“更新代码”、“修复 bug”等模糊描述
- 每次提交**只包含一个相关的修改**，禁止一次提交多个不相关的功能
- 提交前必须在本地通过代码格式检查和单元测试，禁止将问题代码推送到远程

##### 3. Pull Request (PR) 约定

- 每个功能完成后，从个人分支提交 PR 到对应的集成分支
- **PR 标题格式**：`[模块] 功能描述`，例如：`[背包] 实现物品拾取功能`
- **PR 描述必须包含**：功能说明、测试情况、相关 Issue 编号
- 每个 PR 的代码量**不超过 500 行**，超过的必须拆分成多个 PR
- 每个 PR 必须指定**至少 1 名**其他组员作为评审人
- 评审人必须在 **2 小时内**完成评审，给出明确的评审意见（批准/请求修改）
- 提交人必须根据评审意见修改代码，直到评审通过
- 合并前必须同步上游最新代码，解决所有冲突
- 评审通过后，由**评审人合并 PR**，禁止自己合并自己的 PR

##### 4. 代码规范约定

- 后端代码严格遵循 **PEP 8** 规范，使用 `ruff` 进行自动检查和格式化
- 前端代码严格遵循 **ESLint** 和 **Prettier** 规范，提交前自动格式化
- **命名规范**：变量和函数 → 小驼峰式，类名 → 大驼峰式，常量 → 全大写 + 下划线
- 所有函数必须添加**文档字符串**，说明功能、参数和返回值
- 关键逻辑必须添加**行内注释**，说明代码的作用和思路
- **禁止使用魔法值**，所有常量必须定义在单独的配置文件中
- **函数长度不超过 50 行**，超过的必须拆分成多个小函数
- 避免深度嵌套，最多不超过 3 层

##### 5. 沟通与协作约定

- 每日晚上 **9:00** 进行 **15 分钟** 线上站会，每个组员同步：今天做了什么、遇到了什么问题、明天要做什么
- 遇到技术问题先在团队群里讨论，无法解决的再请教老师
- 所有问题和解决方案必须记录在对应的 **GitHub Issue** 中，确保可追溯
- 任务进度延迟必须**提前 1 天**向组长说明，并提出解决方案
- 重大决策通过小组投票决定，少数服从多数
- 鼓励互相帮助，分享技术经验，共同解决问题

#### 约束执行与监督

- 组长负责监督所有约束的执行情况，对违反约束的行为及时提醒和纠正
- 所有组员都有权利和义务监督其他组员遵守约束
- 对于多次违反约束且拒不改正的组员，团队有权向老师反映情况
- 约束的修改必须经过全体组员讨论通过，并记录在案

#### 代码提交流程

所有代码提交必须严格遵循以下标准化流程，确保提交历史清晰可读，便于问题追溯和版本回滚。
**同步上游最新代码**
每次开始新功能开发前，必须先同步对应集成分支的最新代码，避免后续出现大量冲突：
```
git checkout dev/xxx           # 切换到个人分支
git fetch origin               # 获取远程最新代码
git merge origin/backend       # 合并后端集成分支代码（后端开发）
git merge origin/frontend      # 合并前端集成分支代码（前端开发）
```
![PR 评审记录截图 1](https://bee-reg-ab.imagency.cn/e/cce51f0fa48c3156e86c6d8164195bb7.png)
**进行功能开发**
在个人分支上进行开发，每次只完成一个独立的小功能，避免一次提交大量不相关的代码。
**本地代码检查与测试**
提交代码前必须在本地完成以下检查，确保没有明显错误：

- **后端**：运行 `ruff check .` 进行代码格式检查，运行 `pytest` 执行单元测试
- **前端**：运行 `npm run lint` 进行代码格式检查，运行 `npm run test` 执行单元测试
- 所有检查和测试必须**全部通过**，不允许将问题代码推送到远程

**提交代码**
严格采用 **Angular 提交规范**，提交信息格式为：`type(scope): 描述`

- **提交类型**：`feat`(新功能)、`fix`(修复 bug)、`docs`(文档)、`test`(测试)、`ci`(CI/CD)、`chore`(杂项)、`refactor`(重构)
- **作用域**：可选，说明本次提交影响的模块，例如 `backpack`、`player`、`login`
- **描述**：简洁明了地说明本次提交的内容，禁止使用“更新代码”、“修复 bug”等模糊描述

**示例：**
```
git add .
git commit -m "feat(backpack): 实现物品拾取功能"
git commit -m "fix(player): 修复体力值计算错误的bug"
git commit -m "docs(readme): 补充项目部署说明"
```
**推送个人分支到远程**
```
git push origin dev/xxx
```
#### Pull Request (PR) 与代码评审流程

所有代码合并必须通过 PR 进行，严格禁止直接推送到任何保护分支。我们建立了严格的 PR 评审规则，确保每一行代码都经过至少两个人的检查，有效降低了 bug 率，提高了代码质量。
**PR 创建要求**
1. 从个人开发分支提交 PR 到对应的集成分支（前端 → `frontend`，后端 → `backend`）
2. **PR 标题格式**：`[模块] 功能描述`，例如 `[背包] 实现物品排序功能`
3. **PR 描述必须包含以下内容**：
	- **功能说明**：详细描述本次 PR 实现的功能或修复的问题
	- **测试情况**：说明本地测试的结果，包括测试用例和测试步骤
	- **关联 Issue**：使用 `#Issue编号` 关联对应的任务 Issue，例如 `#12`
	- **截图/演示**：如果是 UI 相关的修改，必须附上截图或演示视频
4. 每个 PR 的代码量**不超过 500 行**，超过的必须拆分成多个独立的 PR
5. PR 创建后，自动触发 CI 流水线，执行代码格式检查和单元测试

**代码评审规则**

1. 每个 PR 必须指定**至少一名**其他组员作为评审人：
	- 后端 PR：由胡晨煊和麦振威交叉评审
	- 前端 PR：由余忠祥和翟鹏杰交叉评审
2. 评审人必须在 **2 小时内**完成评审，给出明确的评审意见：
	- **批准 (Approve)**：代码没有问题，可以合并
	- **请求修改 (Request changes)**：代码存在问题，需要修改后重新提交
	- **评论 (Comment)**：提出一些建议性的意见，不影响合并
3. 评审必须覆盖以下四个维度：
	- **正确性**：代码是否实现了预期的功能，是否存在逻辑错误
	- **规范性**：代码是否符合团队制定的代码规范
	- **可读性**：代码是否清晰易懂，是否有必要的注释
	- **性能**：代码是否存在明显的性能问题

**PR 修改与合并流程**

1. 如果评审人提出修改意见，提交人必须根据意见修改代码，并推送新的提交到同一个分支，PR 会自动更新
2. 修改完成后，重新请求评审人进行审核
3. 所有评审意见都得到解决，并且 CI 流水线全部通过后，由**评审人合并 PR** 到集成分支
4. 禁止提交人自己合并自己的 PR

### 代码规范与检查流程

本项目建立了 **“统一规范 + 三重检查”的代码质量保障体系**，完全覆盖课程 **软件代码评审 (15%)** 评分项的所有要求。所有规范由团队共同讨论制定，通过自动化工具强制执行，确保全团队代码风格高度一致，从源头避免不规范代码进入代码库。余忠祥担任代码规范负责人，统筹前后端代码规范的制定、执行与监督。
#### 统一代码规范

我们针对前后端不同技术栈制定了详细的代码规范，所有规范均参考行业最佳实践，并结合本次项目的实际情况进行了调整。
##### 后端 Python 代码规范

后端严格遵循 **PEP 8** 编码规范，并基于 ruff 工具制定了更严格的检查规则：

|规范类别|具体要求|
|:-:|:-:|
|**命名规范**|• 变量名、函数名、模块名：小驼峰式（snake_case） • 类名：大驼峰式（PascalCase） • 常量：全大写 + 下划线分隔 • 禁止使用单字符变量名（循环变量 i/j/k 除外） • 命名必须见名知意，禁止使用拼音和缩写|
|**代码格式**|• 使用 4 个空格缩进，禁止使用 Tab • 每行代码长度不超过 88 个字符 • 函数之间空 2 行，类的方法之间空 1 行 • 导入语句按标准库→第三方库→本地库的顺序排列，组之间空 1 行 • 禁止使用通配符导入（from module import *）|
|**注释规范**|• 所有公共函数和类必须添加文档字符串（docstring），说明功能、参数和返回值 • 关键逻辑必须添加行内注释，说明代码的作用和思路 • 注释必须清晰准确，禁止出现与代码不符的注释 • 禁止注释掉的代码提交到代码库|
|**代码结构**|• 函数职责单一，长度不超过 50 行 • 条件嵌套深度不超过 3 层 • 禁止使用魔法值，所有常量必须定义在单独的配置文件中 • 所有异常必须被捕获并处理，禁止空的 except 块|
##### 前端 TypeScript/Vue 代码规范

前端严格遵循 **Airbnb JavaScript 规范**和 **Vue 3 官方风格指南**，基于 ESLint 和 Prettier 强制执行：

|规范类别|具体要求|
|:-:|:-:|
|**命名规范**|• 变量名、函数名：小驼峰式（camelCase） • 组件名、类名：大驼峰式（PascalCase） • 常量：全大写 + 下划线分隔 • Vue 组件文件名使用大驼峰式，如 `PlayerInfo.vue`|
|**代码格式**|• 使用 2 个空格缩进 • 每行代码长度不超过 100 个字符 • 字符串优先使用单引号，模板字符串使用反引号 • 语句末尾必须加分号 • 对象和数组的最后一个元素后面必须加逗号|
|**Vue 组件规范**|• 使用 `<script setup>` 语法 • 组件 props 必须定义类型和默认值 • 事件名使用短横线命名法（kebab-case） • 模板中指令使用缩写形式（`@` 代替 `v-on`，`:` 代替 `v-bind`） • 组件按 `template` → `script` → `style` 的顺序排列|
|**TypeScript 规范**|• 禁止使用 `any` 类型，必须明确指定所有变量和函数的类型 • 使用接口（interface）定义对象类型 • 使用枚举（enum）定义固定的常量集合 • 函数必须指定返回值类型|
##### 通用规范

- 所有文件使用 **UTF-8** 编码，**LF** 换行符
- 提交信息必须符合 **Angular 提交规范**
- 所有敏感信息（如数据库密码、API 密钥）必须存储在**环境变量**中，禁止硬编码在代码里
- 代码中禁止出现任何个人信息和无关的注释

#### 三重代码检查流程

我们建立了 **“本地 pre-commit 检查 + CI 自动检查 + PR 人工评审”** 的三重检查机制，确保不规范的代码无法进入代码库。
##### 第一重：提交前本地自动检查

我们配置了 Git pre-commit 钩子，在代码提交前自动运行代码检查和格式化，检查不通过则无法提交代码。
**后端检查流程**
1. 运行 ruff 进行代码格式检查和语法错误检测
2. 自动修复可以自动修复的问题
3. 运行单元测试，确保修改不会影响现有功能
4. 所有检查通过后才能提交代码

**前端检查流程**
1. 运行 ESLint 进行代码质量检查
2. 运行 Prettier 自动格式化代码
3. 运行 TypeScript 类型检查
4. 运行前端单元测试
5. 所有检查通过后才能提交代码

##### 第二重：CI 流水线自动检查

每次提交代码到远程仓库或创建 PR 时，GitHub Actions 会自动触发 CI 流水线，再次运行完整的代码检查和测试。
**CI 检查步骤**
1. 拉取最新代码
2. 安装项目依赖
3. 后端：运行 ruff 代码检查
4. 后端：运行 pytest 单元测试
5. 前端：运行 ESLint 代码检查
6. 前端：运行 Vitest 单元测试
7. 前端：运行 TypeScript 类型检查
8. 构建项目，确保代码可以正常编译

**检查结果处理**
- **所有检查通过**：流水线显示绿色通过状态
- **任何一项检查失败**：流水线立即停止，显示红色失败状态，并给出详细的错误信息
- 检查失败的 PR **禁止合并**，必须修复问题后重新提交

#### 测试缺陷管理流程

我们使用 GitHub Issues 进行缺陷管理，所有 bug 都创建为独立的 Issue，确保每个 bug 都有人负责、有记录、有结果。
##### 缺陷分类

按严重程度分为 4 个等级：
- **致命**：导致系统崩溃、数据丢失、无法正常运行的 bug
- **严重**：主要功能模块无法正常运行，影响游戏核心流程的 bug
- **一般**：次要功能模块存在问题，但不影响核心流程的 bug
- **轻微**：界面显示、文字错误等不影响功能使用的 bug

##### 缺陷生命周期

1. **提交**：测试人员发现 bug 后，创建 GitHub Issue，添加 `bug` 标签，注明严重程度、复现步骤和预期结果
2. **分配**：组长将 bug 分配给对应的开发人员
3. **修复**：开发人员修复 bug 后，提交 PR 并关联对应的 Issue
4. **验证**：测试人员验证 bug 是否已修复，验证通过则关闭 Issue
5. **重开**：如果 bug 未修复或引入了新的问题，重新打开 Issue

### 环境与依赖配置问题

#### 后端 Python 依赖版本指定问题

**问题描述**：本地开发环境可正常安装并运行项目依赖，但 GitHub Actions CI 流水线执行依赖安装步骤时失败，无法正常安装 Pydantic 相关库，流水线中断。
**原因分析**：项目 `requirements.txt` 中对 Pydantic 采用**精确版本锁定（`pydantic==2.0.0`）**，该固定版本对系统环境、Python 版本及配套依赖限制严苛。CI 流水线的运行环境、底层组件与本地开发环境存在差异，严格锁定单一版本导致依赖解析、兼容校验失败，最终无法完成安装。
**解决方案**：
1. 修改 `requirements.txt` 文件，将 Pydantic 版本约束从精确匹配 `==` 调整为兼容范围 `>=`，放宽版本限制，适配 CI 运行环境。
2. 本地重新执行依赖安装测试，保证修改后本地功能不受影响。
3. 在 CI 流水线中添加依赖缓存步骤，优化后续构建速度。

**预防措施**：区分核心强依赖与兼容型依赖，对兼容性较好的通用库合理使用版本范围约束；上线 CI 前，优先在模拟流水线环境验证依赖安装效果；定期统一维护项目依赖清单，保障本地、测试、线上环境整体兼容。

##### 前端 Node.js 版本不兼容

**问题描述**：部分组员本地使用 Node.js 18 版本运行 `npm run dev` 时，出现 `SyntaxError: Unexpected token '??='` 语法错误，而使用 Node.js 22 的组员可正常运行。
**原因分析**：项目使用 Vite 5.2 作为构建工具，该版本最低要求 Node.js 20.0 以上，Node.js 18 不支持空值赋值运算符 `??=` 等 ES2021 新语法。
**解决方案**：
1. 在 `package.json` 中添加 `engines` 字段，明确指定 Node.js 版本要求
2. 在 README 文档中添加开发环境准备指南，要求所有组员统一使用 Node.js 22 LTS 版本
3. 在 CI 流水线中固定使用 `node-version: '22'`

**预防措施**：项目启动时首先统一所有组员的开发环境版本；在配置文件中明确指定最低支持的运行环境；新工具引入前先验证版本兼容性。

#### 前后端开发与联调问题

##### 跨域请求被浏览器阻止

**问题描述**：前端本地开发时，调用后端登录接口出现 CORS 错误：`Access to XMLHttpRequest at 'http://localhost:8000/api/player/login' from origin 'http://localhost:5173' has been blocked by CORS policy`。
**原因分析**：后端 FastAPI 未正确配置 CORS 中间件，默认拒绝所有跨域请求，未添加前端本地开发地址 `http://localhost:5173` 到允许源列表。
**解决方案**：在后端 `main.py` 中添加 CORS 中间件，区分开发环境和生产环境配置允许源。
**预防措施**：项目初始化时首先配置 CORS 中间件；严格区分开发与生产环境的安全策略；联调前先使用 Apifox 测试接口跨域访问是否正常。

##### 前后端接口字段名不一致

**问题描述**：注册功能联调时，前端发送的请求体包含 `playerName` 字段，但后端始终返回“用户名不能为空”的错误。

3. 使用 Apifox 的自动化测试功能，批量验证所有接口的参数正确性

**预防措施**：后端接口开发完成后必须先更新接口文档；前端开发前必须仔细核对接口文档；联调前先通过 Apifox 单独测试每个接口，确保参数和返回值符合预期。

#### 版本控制与团队协作问题

##### 多人修改同一文件导致严重冲突

**问题描述**：麦振威和胡晨煊同时开发玩家模块的不同功能，都修改了 `player_service.py` 文件中的 `movePlayer` 函数，提交 PR 时出现 30+ 行代码冲突，Git 无法自动合并。
**原因分析**：任务拆分不够细致，导致两人同时修改同一个核心函数的不同部分；且两人在开发前未同步进度，都基于旧版本代码进行开发。
**解决方案**：
1. 两人共同对比冲突代码，逐行分析保留双方的正确修改
2. 解决冲突后在本地进行完整的功能测试，确保没有引入新 bug
3. 重新提交 PR 进行代码评审

**预防措施**：拆分任务时尽量避免多人同时修改同一个文件；如果必须修改同一文件，提前在团队群里沟通，分时段开发；每次提交 PR 前先执行 `git pull --rebase` 同步上游最新代码，在本地解决冲突后再推送。

##### 敏感信息误提交到代码库

**问题描述**：翟鹏杰在提交配置文件时，不小心将包含数据库明文密码的 `.env` 文件推送到了远程仓库。
**原因分析**：项目初始化时未及时配置 `.gitignore` 文件，导致 Git 将 `.env` 识别为普通文件；且开发人员提交前未仔细检查提交的文件列表。

4. 将所有敏感信息存储到 GitHub Secrets 中，代码中仅通过环境变量引用

**预防措施**：项目创建后首先编写完善的 `.gitignore` 文件；所有敏感信息禁止硬编码在代码中；提交代码前必须使用 `git status` 检查待提交文件列表。

#### CI/CD 与部署问题

##### GitHub Actions 无法连接至 Replit 平台

**问题描述**：CD 流水线执行后端部署步骤时，抛出 `ssh: handshake failed: ssh: unable to authenticate, attempted methods [none password], no supported methods remain` 错误，无法连接到后端服务 Replit 服务器。
**原因分析**：在 GitHub Secrets 中输入服务器密码时，误输入了多余的空格，导致 SSH 密码认证失败。
**解决方案**：
1. 重新在 GitHub 仓库的 `Settings → Secrets and variables → Actions` 中输入正确的服务器密码
2. 在 CD 流水线中添加 SSH 连接测试步骤，提前验证认证是否成功
3. 改用 SSH 密钥认证方式替代密码认证，提高安全性

**预防措施**：配置 Secrets 时仔细核对输入内容，避免多余的空格和换行；优先使用 SSH 密钥认证而非密码认证；在流水线中添加前置检查步骤，验证所有配置的正确性。

##### 后端服务启动时端口被占用

**问题描述**：第一次部署后端服务时，启动脚本执行失败，错误信息为 `OSError: [Errno 98] Address already in use: 0.0.0.0:8080`。
**原因分析**：启动脚本中没有杀死旧进程的步骤，上一次部署失败后残留的后端进程占用了 8080 端口。
**解决方案**：修改 `start.sh` 启动脚本，在启动新服务前强制杀死占用 8080 端口的所有进程。
**预防措施**：所有服务启动脚本必须包含旧进程清理步骤；添加端口检测逻辑，确认端口可用后再启动新服务；部署完成后执行健康检查，验证服务是否正常运行。

#### 测试与数据一致性问题

##### 并发情况下物品被重复拾取

**问题描述**：使用 wrk 进行性能测试时，模拟 10 个并发用户同时拾取同一个物品，发现所有用户都成功拾取了该物品，物品被重复获取，导致数据不一致。
**原因分析**：拾取物品的接口没有添加并发控制，多个请求同时读取到物品存在的状态，然后同时执行删除房间物品和添加背包物品的操作，最终所有请求都执行成功。
**解决方案**：在数据库事务中使用行级锁，确保同一时间只有一个请求能够操作该物品记录。
**预防措施**：所有涉及数据修改的接口必须考虑并发场景；使用数据库事务和锁机制保证数据一致性；上线前必须进行并发性能测试，提前发现数据一致性问题。

#### 问题总结与经验收获

本次项目中遇到的所有问题，都是软件开发过程中非常典型且高频出现的问题。通过解决这些问题，我们团队获得了以下宝贵的经验：

5. **团队沟通至关重要**：很多问题的根源都是沟通不畅，提前沟通、及时同步可以避免大部分不必要的冲突和返工。

### 问题解决与经验教训

在项目开发过程中，我们遇到了环境配置、代码开发、团队协作、部署运维等多个维度的典型问题。通过解决这些问题，我们积累了宝贵的实战经验：
1. **环境一致性是基础**：“本地能跑线上跑不了”是最常见也最头疼的问题。必须通过版本锁定、环境配置统一等手段保证开发、测试、生产环境的一致性。
2. **规范先行是效率保障**：提前制定代码规范、接口规范、提交规范、分支规范，可以大幅减少后续的沟通成本和冲突问题。很多问题的根源都是没有统一的规范。
3. **自动化是质量底线**：将重复的代码检查、测试、部署工作交给机器完成，不仅可以提高效率，还可以避免人为错误。能自动化的一定要自动化。
4. **并发问题不可忽视**：所有涉及数据修改的操作都必须考虑并发场景，使用数据库事务和锁机制保证数据一致性。上线前一定要进行并发性能测试。
5. **团队沟通至关重要**：很多问题的根源都是沟通不畅。提前沟通、及时同步、透明公开，可以避免大部分不必要的冲突和返工。

### 不足与未来改进方向

虽然本次项目取得了不错的成果，但我们也认识到存在一些不足，需要在未来的学习和实践中不断改进：
1. **单元测试覆盖率有待提高**：目前核心功能覆盖率达到了 85%，但边缘功能和异常处理的覆盖率还不够高。未来应该进一步提高单元测试覆盖率，实现测试驱动开发 (TDD)。
2. **没有使用容器化技术**：本次项目没有使用 Docker 进行容器化部署，环境配置仍然比较繁琐。未来应该引入 Docker 和 Docker Compose，实现“一次构建，到处运行”。
3. **实时通信能力不足**：目前多人在线功能是通过前端轮询实现的，实时性较差。未来可以引入 WebSocket 技术，实现真正的实时通信。
4. **性能测试不够全面**：本次只进行了简单的接口性能测试，没有进行压力测试和负载测试。未来应该使用更专业的性能测试工具，模拟更高的并发场景。
5. **文档可以更完善**：虽然我们编写了基本的项目文档，但 API 文档、用户手册、运维手册等还不够详细。未来应该更加重视文档的编写，提高项目的可维护性。
